Git学习笔记


1. 基本流程

git clone address把github上的代码clone到本地文件夹下–>git add .放到暂存区–>git commit -m”备注信息”提交记录–>git push(分支)

2. 基本命令

git add files 把当前文件放入暂存区域。

git commit 给暂存区域生成快照并提交。

git reset -- files 用来撤销最后一次git add files，你也可以用git reset 撤销所有暂存区域文件。

git checkout -- files 把文件从暂存区域复制到工作目录，用来丢弃本地修改。

可以用 git reset -p, git checkout -p, or git add -p进入交互模式。

也可以跳过暂存区域直接从仓库取出文件或者直接提交代码。

  • git commit -a 相当于运行 git add 把所有当前目录下的文件加入暂存区域再运行。
  • git commit files 进行一次包含最后一次提交加上工作目录中文件快照的提交。并且文件被添加到暂存区域。
  • git checkout HEAD – files 回滚到复制最后一次提交。

3. diff、commit、checkout命令

查看两次提交之间的变动：git diff file
git会使用与当前提交相同的父节点进行一次新提交，旧的提交会被取消。 git commit --amend
checkout命令用于从历史提交（或者暂存区域）中拷贝文件到工作目录，也可用于切换分支。git checkout main

4. reset、merge命令

reset命令把当前分支指向另一个位置，并且有选择的变动工作目录和索引。也用来在从历史仓库中复制文件到索引，而不动工作目录。

merge 命令把不同分支合并起来。合并前，索引必须和当前提交相同。如果另一个分支是当前提交的祖父节点，那么合并命令将什么也不做。 另一种情况是如果当前提交是另一个分支的祖父节点，就导致fast-forward合并。指向只是简单的移动，并生成一个新的提交。

5. Cherry Pick、Rebase命令

cherry-pick命令”复制”一个提交节点并在当前分支做一次完全一样的新提交。

Rebase衍合是合并命令的另一种选择。合并把两个父分支合并进行一次提交，提交历史不是线性的。衍合在当前分支上重演另一个分支的历史，提交历史是线性的。 本质上，这是线性化的自动的 git rebase --interactive

6. 提交记录

Git 仓库中的提交记录保存的是你的目录下所有文件的快照，就像是把整个目录复制，然后再粘贴一样，但比复制粘贴优雅许多！

提交记录尽可能地轻量，因此在你每次进行提交时，它并不会盲目地复制整个目录。条件允许的情况下，它会将当前版本与仓库中的上一个版本进行对比，并把所有的差异打包到一起作为一个提交记录。

通过git log查看哈希表

7. git branch

Git 的分支也非常轻量。它们只是简单地指向某个提交纪录 —— 仅此而已。

这是因为即使创建再多的分支也不会造成储存或内存上的开销，并且按逻辑分解工作到不同的分支要比维护那些特别臃肿的分支简单多了。

在将分支和提交记录结合起来后，我们会看到两者如何协作。现在只要记住使用分支其实就相当于在说：

“我想基于这个提交以及它所有的 parent 提交进行新的工作。”

现在咱们告诉 Git 我们想要切换到新的分支上

git checkout <name>

接下来的git commit 就会在该分支上进行。

8. 创建分支并切换过去

git branch yumi
git checkout yumi

等价于
git checkout -b <yumi>

9. 合并分支merge

接下来咱们看看如何将两个分支合并到一起。就是说我们新建一个分支，在其上开发某个新功能，开发完成后再合并回主线。

咱们先来看一下第一种方法 —— git merge。在 Git 中合并两个分支时会产生一个特殊的提交记录，它有两个 parent 节点。翻译成自然语言相当于：

“我要把这两个 parent 节点本身及它们所有的祖先都包含进来。”

10. git rebase合并

Rebase 实际上就是取出一系列的提交记录，“复制”它们，然后在另外一个地方逐个的放下去。

Rebase 的优势就是可以创造更线性的提交历史，这听上去有些难以理解。如果只允许使用 Rebase 的话，代码库的提交历史将会变得异常清晰：

  1. git checkout yumi当前节点

  2. git rebase a将当前节点复制到a节点下面（作为a的子节点）

  3. git checkout aa节点

  4. git rebase yumi合并

19. git push

git push 和 git push origin main 之间的对应关系如下：

组成部分
<remote>：
这是你要推送到的远程仓库的名称。默认情况下，Git 在克隆仓库时会将远程仓库命名为 origin。你也可以添加其他远程仓库，例如 upstream。

<branch>：
这是你要推送的本地分支的名称。它对应于远程仓库中的分支。

示例解析

git push origin main：

origin：表示远程仓库的名称，通常是默认配置的远程仓库。
main：表示你要推送的本地分支的名称。在这个例子中，推送的是名为 main 的分支。

20. git push

在 GitHub 上处理别人的库（通常是指开源项目）时，你不能直接推送到他们的主库。相反，通常采用以下步骤：

  1. Fork（分叉）库
    Fork 仓库：
    在 GitHub 上，找到你想要贡献的库，点击右上角的 “Fork” 按钮。这会在你的 GitHub 账户下创建该库的副本。
  2. Clone（克隆）库
    克隆你的 Fork：
    在你的本地机器上，使用克隆命令将你的 Fork 下载到本地：
    bash

复制
git clone https://github.com/你的用户名/库名.git
例如：
bash

复制
git clone https://github.com/yourusername/repo.git
3. 创建分支
创建新分支：
在本地仓库中，为了进行更改，建议你创建一个新分支：
bash

复制
git checkout -b feature-branch
4. Make Changes（进行更改）
进行更改：
在新分支上进行更改，完成后使用 git add 和 git commit 提交更改。
5. Push Changes（推送更改）
推送到你的 Fork：
将你在本地的更改推送到你的 GitHub Fork：
bash

复制
git push origin feature-branch
6. Create a Pull Request（创建拉取请求）
创建 Pull Request：
登录 GitHub，进入你的 Fork 页面，点击 “Pull Requests” 标签，然后点击 “New Pull Request”。
选择你的分支并提交请求，向原始仓库的维护者请求合并你的更改。
7. Sync with Original Repository（同步原始仓库）
如果原始仓库有更新，你可能需要将这些更新合并到你的 Fork：

添加原始仓库为远程：
bash

复制
git remote add upstream https://github.com/原作者/库名.git
拉取更新：
bash

复制
git fetch upstream
git checkout main # 切换到主分支
git merge upstream/main
推送更新到你的 Fork：
bash

复制
git push origin main
总结
你不能直接推送到别人的库，但可以通过 Fork、克隆、创建分支、推送到你的 Fork 并创建 Pull Request 来贡献更改。
记得保持你的 Fork 与原始仓库同步，以便于合并最新的更改。
